# DSI Microscope — Project Overview & Code Architecture

Control software for the **Institut Fresnel event-based Dynamic Speckle Illumination (event-DSI) microscope**. It is a PyQt6 desktop application that drives the full optical setup and reconstructs optically-sectioned wide-field fluorescence images.

>
> For the formal math — every equation behind acquisition and reconstruction, with rendered symbols — see [MATHEMATICS.md](MATHEMATICS.md).

---

## 1. The scientific idea

Wide-field fluorescence microscopy is fast and cheap but has **no depth discrimination**: light from out-of-focus planes blurs into the image and destroys contrast. **Dynamic Speckle Illumination (DSI)** recovers optical sectioning without a confocal pinhole or scanning.

The principle:

- The sample is illuminated with a **fluctuating random speckle pattern** (here generated by a zwitterion-doped liquid-crystal diffuser driven by an AC field).
- The detection point-spread function is sharpest **at the focal plane** and diverges away from it. So **in-focus** speckle grains are sharp and produce large pixel-to-pixel intensity fluctuations as the speckle decorrelates, while **out-of-focus** signal is spatially averaged and barely fluctuates.
- Computing the **per-pixel standard deviation** (or fluctuation) over a stack of many speckle frames therefore yields an image dominated by in-focus signal → optical sectioning (~2 µm axial). The **per-pixel average** of the same stack is the conventional widefield-equivalent image.

This project implements **two acquisition modalities** of that idea:

| Modality | Detector | How sectioning is computed | Output |
|----------|----------|----------------------------|--------|
| **Traditional DSI** | Hamamatsu ORCA-Fusion BT sCMOS (frame-based) | Software: per-pixel std-dev across a frame stack (`compute_dsi_images`) | Average (widefield) + std-dev (sectioned) image |
| **Event-DSI** | Prophesee EVK4 event camera (neuromorphic) | Hardware: the sensor only fires events on intensity *changes*, so accumulating events ≈ fluctuation map, with virtually no post-processing | 2D event-count image |

The **event-DSI** advantage (per the reference papers): the sensor reacts to intensity *variations* by design, so it natively performs the operation DSI needs. Data shrinks from gigabytes of frames to tens of kB of events, with little to no compute. The trade-offs handled in code are event-camera artifacts: **"crazy pixels"** (a few pixels that self-trigger enormously) and **sensor inhomogeneity** (corrected by smoothing).

### The instruments

- **532 nm CW laser** → beam expanded onto a **liquid-crystal (LC) diffuser** → generates random speckle. The LC's electrohydrodynamic instabilities (and thus the speckle decorrelation time) are driven by a **square-wave AC field** from the AWG.
- **Siglent SDG6022X AWG** — drives the LC; two independent channels (CH1/CH2). Frequency/amplitude set the speckle dynamics.
- **40× / 0.65 NA objective** on a **PI E-709 piezo Z-stage** for focus and Z-stacks.
- **Detection path** splits to either the **ORCA sCMOS** or the **Prophesee EVK4** event camera.

## 2. Architectural principle: strict layering

The codebase is organized into four layers with a **one-directional dependency rule**, so the physics math stays testable and the hardware stays swappable:

```
        ┌─────────────────────────────────────────────┐
        │  ui/   (PyQt6 — windows, widgets, threads)   │
        │  main_window · widgets · orchestrator        │
        └───────────────┬──────────────┬──────────────┘
                        │              │
            ┌───────────▼──────┐   ┌───▼──────────────┐
            │  hardware/       │   │  core/           │
            │  SDK wrappers +  │──▶│  pure NumPy /    │
            │  QThread workers │   │  OpenCV / SciPy  │
            └───────────┬──────┘   └───┬──────────────┘
                        │              │
                  ┌─────▼──────────────▼─────┐
                  │  config.py               │
                  │  paths · constants · CSS │
                  └──────────────────────────┘
```

**Rules enforced by the module docstrings:**

- `config.py` imports **nothing** heavy (no PyQt6, no SDKs) — safe to import everywhere.
- `core/` imports **no PyQt6 and no hardware SDK** — pure, side-effect-free math (apart from explicit `save_*` helpers). This is the optical-sectioning physics, unit-testable in isolation.
- `hardware/` wraps vendor SDKs and owns the threading. It depends on `core` (to do the math) and `config`, never on `ui`.
- `ui/` is the only layer that instantiates workers and wires Qt signals. The Z-stack **orchestrator** lives here (not in `hardware/`) deliberately, because it composes *two* instruments plus `core`, and putting it in `hardware/` would create a `hardware → core → hardware` tangle.

### Threading model

All blocking instrument I/O runs on **`QThread` workers** so the GUI never freezes. Workers communicate back to the main thread **only** via Qt signals (`status_update`, `image_ready`/`frame_ready`, `z_profile_update`, `position_update`, `error_signal`, `finished_signal`). Every worker exposes a cooperative `stop()` that flips an `_is_running` flag checked inside its loop, and guarantees device teardown via `try/finally`.

### Graceful degradation

Every SDK import (`dcam`, `metavision_*`, `pipython`) is wrapped in `try/except ImportError` with an `*_AVAILABLE` flag. The GUI builds and runs on a dev machine with **no cameras/drivers**; attempting an action without the SDK shows a clear dialog (with setup hints) instead of crashing.

## 3. File-by-file guide

### Entry point
- **`main.py`** — builds the `QApplication`, applies the stylesheet, shows `MainWindow`. ~20 lines.

### `config.py`
Dependency-light constants:
- **SDK path** — `HAMAMATSU_SDK_PATH` (where `dcam.py` lives), overridable by env var.
- **PI stage** — controller name, serial, RS-232 port/baud, auto-zero flag. Most are env-overridable so the same checkout runs on laptop and lab PC.
- **ORCA / DCAM** — property ids (exposure, readout speed, binning, trigger source/mode, defect correct, and the **subarray** HPOS/HSIZE/VPOS/VSIZE/MODE ids); sensor size `ORCA_SENSOR_WIDTH/HEIGHT = 2304`; `*_OPTIONS` maps ({label: dcam enum value}) surfaced in the UI combos; and **`ORCA_ROW_READOUT_US`** — the per-row readout time `1H` for each scan speed, taken directly from the camera manual (Fast 4.86765 µs, Standard 18.64706 µs, Ultra Quiet 80.0 µs). The free-running frame time is `(Vn+1)·1H`, so reducing subarray **height** is what raises the framerate.
- **EVK4** — event-rate-controller cap, display fps, crazy-pixel percentile.
- **FOV matching** — `EVK4_TO_ORCA_AFFINE`, the calibrated 2×3 affine mapping full-sensor EVK4 (x, y) → full-sensor unbinned ORCA (x, y), measured by the 2026-07-10 masked-NCC bead registration (`ports_100`: θ = 317°, scale = 0.745 = the 4.86/6.5 µm pixel-pitch ratio; only the *translation* drifts on realignment). Overridable without code changes by pointing `DSI_FOV_AFFINE_PATH` at a `registration.json` (`DSI_FOV_AFFINE_KEY` picks the entry). `FOV_MATCH_PATH` — where the UI persists the last user-confirmed matching crop (per-user app-data).
- **`STYLESHEET`** — the dark-theme Qt CSS.

### `core/` — pure processing (no Qt, no SDKs)
**`image_processing.py`** — public API re-exported by `core/__init__.py`:

*ORCA / DSI:*
- `compute_dsi_images(stack, roi=None)` — **the active single-plane DSI reconstruction**. Returns the per-pixel **average** (widefield-equivalent) and **standard-deviation** (optically-sectioned) float32 images across the (N,H,W) stack. Used by both the ORCA single-Z acquire and the Z-stack. **Memory-bounded:** mean and variance are accumulated over the frame axis in float64 **chunks** (a two-pass population variance, `ddof=0`, matching the old `np.std`), instead of materializing `stack.astype(float32)` and `np.std`'s internal deviation array — each the full N×H×W size. The working set is a few H×W planes (`_DSI_CHUNK_BYTES ≈ 128 MB`), so peak RAM is independent of N (this is what stops a big stack from swapping the machine).
- `process_dsi(stack, roi)` — *legacy* consecutive-difference fluctuation estimator `sqrt(Σ(frame[i+1]−frame[i])²)`, returning a scalar focus metric + 8-bit preview. Still exported, but **no current caller** drives the pipeline with it (the Z-stack profile value is `np.sum(std_img)` from `compute_dsi_images`). Keep for reference / re-processing experiments.
- `normalize_to_8bit(img)` — min-max → uint8 for display/preview.
- `crop_to_roi(img, roi)` — slice a 2D/3D image to an `x_min/x_max/y_min/y_max` window, clamped to the frame (identity for `None`/full-frame). The EVK4 software crop for both the live BGR preview and the accumulated event image.
- `save_dsi_results(avg, std, dir, name)` — writes `<name>_average` and `<name>_dsi` as full-precision `.mat` **and** 8-bit `.tif`. **Every generated file starts with the user's filename base** — descriptors are suffixes, not prefixes — so all of a run's outputs sort together.
- `scale_16bit_image(data)` — 16→8-bit live-view scaling via a **fixed** down-shift (full 16-bit range → 8-bit). Deliberately *not* a per-frame auto-stretch: display brightness tracks the real signal, so a shorter exposure looks dimmer (matching the vendor software). The old auto-stretch-by-frame-max inverted that — dimmer frames got a bigger multiplier, making shorter exposures look brighter. Display only; raw data and DSI math are untouched.

*Raw / volume persistence (TIFF):*
- `save_raw_stack_tiff(stack, dir, name, roi=None, plane=None)` — one plane's raw 16-bit frames as a multi-page TIFF (`<name>_raw_stack.tif`), native bit depth preserved. The user's filename base leads the name. Used by the single-Z acquire **and** by the Z-stack, which passes `plane=step` so each plane writes a **separate** z-indexed file (`<name>_raw_stack_zNNN.tif`) — the downstream MATLAB algorithms (e.g. **RIM**) consume the planes individually, not as one combined volume.
- `save_volume_tiff(volume, dir, name, kind)` — a (Z,H,W) depth stack as a multi-page TIFF (float32 → 32-bit, no per-slice normalization).
- `save_axial_sectioning_plot(z_positions, intensities, dir, name, kind)` — the **axial-sectioning profile** (paper Fig. 3a): per-plane mean sectioned-image intensity vs axial position, Gaussian-fitted (`scipy.optimize.curve_fit`) to extract the **FWHM = axial sectioning**. Always writes `<name>_axial_profile_<kind>.csv` (z, raw + peak-normalized intensity, fit params); writes the PNG only if **matplotlib** is installed (lazy OO `FigureCanvasAgg` import, thread-safe, no window). Returns `(fwhm, csv_path, png_path)`. Helpers `_gaussian`, `_fit_axial_gaussian`, `_save_axial_figure` degrade to data-only on any failure (no SciPy/matplotlib, <4 planes, non-convergence).
- `save_axial_average_plot(z_positions, intensities, dir, name)` — **companion profile of the *average* (widefield) image** vs axial position. Unlike the sectioned image, the average collects out-of-focus light at every plane, so its intensity is essentially **flat** across z — it has *no* optical sectioning. It is therefore fitted with a **straight line** (`np.polyfit`, no SciPy needed) rather than a Gaussian; the near-zero slope quantifies the absence of sectioning. Same CSV-always / PNG-if-matplotlib contract: writes `<name>_axial_average.csv` + `.png`. Returns `(slope, csv_path, png_path)`. Helpers `_fit_axial_line`, `_save_average_figure`. **ORCA-only** (the event camera produces no average image).
- `_write_multipage_tiff` / `_prepare_raw_frames` — shared internals (ROI crop + native-depth coercion).

*Logging:*
- `save_parameter_log(dir, name, sections)` — human-readable `<name>_parameters.txt` from an ordered `{section: {key: value}}` dict.

*EVK4:*
- `accumulate_event_frame(chunks, w, h)` — sum events per pixel via `np.add.at` (SDK-agnostic; iterator injected by hardware).
- `filter_crazy_pixels(img)` — zero hot self-triggering pixels above a percentile.
- `apply_smoothing(img)` — 5×5 Gaussian (event-sensor inhomogeneity).
- `save_mat_tif(img, dir, name)` — final EVK4 image as `.mat` + `.tif`.
- `map_evk4_window_to_orca(affine, evk4_roi, orca_sensor, align=4)` — maps an EVK4 crop window through the calibrated EVK4→ORCA affine and returns `(crop, corners, clipped)`: the matching ORCA crop (the **bounding box** of the ~43°-rotated footprint, expanded outward to the 4-px DCAM grid and clamped to the sensor), the four mapped footprint corners (for drawing the true rotated outline), and whether clamping cut into the footprint. Pure NumPy; drives the UI's FOV-matching button.
- `register_evk4_to_orca(orca_img, evk4_img, seed_affine=None, status=None)` — **live re-measurement of the EVK4→ORCA registration** (the in-app port of the 2026-07-10 `step2_register.py`): reduces both full-sensor images to smoothed blob maps (`_blobmap` — the smoothing is what tolerates the ports' small focal-plane offset) and matches by masked NCC (`_masked_ncc`, `cv2.matchTemplate` + mask) over flip/rotation/scale/translation, coarse (¼ res) → refine (½ res) → local full-res translation polish. Seeded by the current affine it searches only θ ± 15° / scale ± 0.06 (fast); if the seeded match is weak (NCC < 0.35) it automatically widens to a full-circle search (camera remounted). Returns `(affine, ncc_score, {flip, theta, scale})`. Verified on synthetic ground truth to < 5 px corner error. `decompose_fov_affine(A)` extracts (flip, θ_cv2, scale); `compose_registration_overlay(orca, evk4, A)` builds the green-ORCA/magenta-EVK4 RGB check image used as the preview background.

### `hardware/` — SDK wrappers + worker threads
- **`orca_camera.py`** — `OrcaWorker(QThread)` for the Hamamatsu ORCA via DCAM. Two modes:
  - **live** — continuous 3-frame-ring-buffer focus loop. Supports **live parameter updates** via `apply_params()`: on the next frame it stops capture, re-applies exposure/mode/ROI, and restarts (a brief blink, far cheaper than a worker restart).
  - **acquire** — single-Z DSI: grabs N speckle frames into a buffer, then `compute_dsi_images` → saves the avg + std images, optionally the **raw 16-bit stack TIFF**, and a parameter log.
  - `_apply_camera_settings` pushes readout speed / binning / trigger / defect-correct (rejections surface as status warnings, never abort). `_apply_subarray` configures the **DCAM hardware subarray (ROI)**: MODE must be OFF before changing position/size; HPOS/HSIZE/VPOS/VSIZE must be multiples of 4 and within the sensor. **If the camera rejects the subarray, it falls back to a software crop** (`_hw_roi_active=False`) — correct image, but no framerate gain. Hardware subarray already delivers the cropped frame, so software crop is only applied when the ROI was rejected.
- **`event_camera.py`** — `CameraWorker(QThread)` for the Prophesee EVK4 (Metavision SDK). Sets biases (`bias_fo`, `bias_hpf`, `bias_diff_on/off`), enables the event-rate controller, streams events into a `PeriodicFrameGenerationAlgorithm` for live view, supports live bias updates via `apply_biases()` and live re-cropping via `apply_roi()`, and in *acquire* mode reconstructs/filters/smooths/saves the final 2D image by delegating to `core`. The raw event stream (`<filename>.raw`) is **always saved** alongside the `.tif`/`.mat` (it is the full event stream the 2D image is derived from; there is no toggle). **ROI cropping:** the event stream is always full-sensor, so the worker **software-crops** every live frame and the accumulated image to the ROI window (`core.crop_to_roi`); in *acquire* mode (fixed window) it also sets a **best-effort hardware ROI** via the module-level `apply_event_roi(device, roi)` to cut the event rate — fully guarded across SDK versions, falling back to software-crop-only (mirrors the ORCA's "hardware subarray, else software crop"). Live mode keeps the full sensor streaming so the crop can be dragged to any sub-window. Raw logging is always stopped in a `finally` (calling stop without a prior `log_raw_data` can crash the native lib). Also exports `apply_event_roi` / `initiate_device` / `EventsIterator` / `METAVISION_AVAILABLE` for the orchestrator to reuse.
- **`stage_control.py`** — `StageController` owns the PI `GCSDevice` handle and connection lifecycle: tries **native USB** first, then **RS-232** (forced port or auto-scan, FTDI/US232R adapters probed first). The E-709 has **no reference switches** → no FNL/FRF; it just enables the servo (`SVO`) for closed-loop, optional auto-zero (`ATZ`). Reads position unit + travel limits (`qPUN`/`qTMN`/`qTMX`) best-effort. `PIMoveWorker(QThread)` offloads blocking `MOV` + `waitontarget`, always emitting `finished_signal` (even on error) so the UI re-enables.
- **`awg_control.py`** — `AWGController`, a thin PyVISA/SCPI wrapper for the Siglent AWG. Lists resources, connects, sets per-channel (`C1`/`C2`) waveform/freq/amp, toggles output, and on `close()` forces both outputs off. No Qt. **Graceful degradation:** `pyvisa` is import-guarded (`PYVISA_AVAILABLE`) and the VISA `ResourceManager` is created lazily (not in `__init__`), so a machine with no VISA backend still builds the GUI (the AWG picker just comes up empty) instead of crashing at startup — mirroring the `pipython`/`dcam`/`metavision` guards. **Amplitude / load:** `set_params` sets the output load (`Cx:OUTP LOAD,HZ`, high-Z for the LC cell) *before* the `BSWV` command, because the Siglent interprets the requested Vpp relative to the load (a 50 Ω load would halve it). The UI (`AWGWidget`) pushes the displayed parameters to the instrument **on connect** and re-applies a channel's params **before enabling its output**, so the hardware always matches the UI values recorded in the acquisition parameter log (the log reports the UI state).

### `ui/` — presentation & coordination
- **`main_window.py`** — `MainWindow`. The app is **Z-Stack-only**: it builds **two tabs**, **Z-Stack (ORCA)** and **Z-Stack (EVK4)** — one per detector, each with full camera parameters, a live preview, the crop tool, and the automated 3D stack for that camera. The shared instruments live in **collapsible left-panel sections** (`btn_awg_toggle`/`awg_widget`, and `btn_pi_toggle`/`pi_stage_widget` — the PI Z-stage, used by both tabs; both start **collapsed**) so either tab can reach them. It instantiates all workers + the Z-stack orchestrator, routes every worker's `status_update`/`error_signal` to one status bar, manages button enable/disable via `_refresh_buttons`, and renders frames (NumPy → `QImage` → `QPixmap`, grayscale or BGR888). Also owns:
  - **Dual-camera live view** — because the detection path uses a **beamsplitter** (both detectors see the sample at once), the feed panel holds **two independent `VideoFeedLabel`s** (ORCA + EVK4), each fed by its own slot (`update_orca_image` / `update_evk4_image`) so the two streams never overwrite each other. A **Display** selector (`combo_feed_view`, defaults to ORCA-only; **follows the active tab** via `_on_tab_changed` unless set to Both) shows one camera at a time or **both stacked**, and a **Start/Stop Both Cameras** button (`btn_both_live` → `_toggle_both_live`) starts/stops the two live feeds together — placed in the shared feed panel because it acts on both at once. It reuses the per-camera `start_orca_live`/`start_evk4_live` paths (starting only the camera not already running) and flips the view to *Both (stacked)*; `_sync_both_live_button` keeps its label correct even when feeds are toggled from their own tabs. The ORCA and EVK4 live feeds run on separate workers (`orca_worker`/`evk4_worker`)/threads/SDKs, so both can stream simultaneously; the shared `_render_to_label` does the NumPy→pixmap scaling. The crop tool targets whichever camera the active tab shows (`_active_crop_target`/`_active_crop_label`/`_active_crop_frame`), and a tab change cancels an in-progress crop (`_cancel_crop_mode`) so its overlay can't strand on the wrong feed. (Simultaneous *acquisition* with timestamp-aligned streams is not implemented — this is live view only.)
  - **Acquisition time estimates** per tab — a **cold-start physics model** (`_orca_zstack_predicted_s`/`_evk4_zstack_predicted_s`) sums camera init plus, per plane, motor move/settle, capture (`N·frame_time`), the **per-plane DSI reconstruction** (`OrcaParamsWidget.estimated_compute_s`), and raw-stack disk write (constants in `config.py`: `ORCA_CAMERA_INIT_S`, `ORCA_PLANE_OVERHEAD_S`, `EVK4_PLANE_OVERHEAD_S`, `ORCA_DSI_PROCESS_PIXELS_PER_S`, `ZSTACK_DISK_BYTES_PER_S`). The per-plane DSI compute (~5–6 s/plane at full sensor) was the big originally-missing cost.
  - **Self-calibrating estimates (learning) — shown as a range** — because per-plane overheads are machine-specific *and highly variable* (the EVK4's per-plane reconstruction cost — decode + accumulate + xytp save — scales with the event count, so wall-clock swings 2–3× with the biases even for identical nominal settings), the cold-start estimate is turned into a **(typical, slow) range** rather than one number: `_calibrate_range` scales the raw estimate by the **50th and 90th percentiles** of past `actual/predicted` ratios for that type (`orca_zstack`/`evk4_zstack`), and `_fmt_range` renders "≈ 3 min 45 s – 5 min 28 s" (collapsing to a single value when there's no history or negligible spread). Records whose actual time is below **half the cold estimate** are treated as aborted/impossible and dropped from the ratios, so a stopped run can't drag the calibration down (belt-and-suspenders with the `_acq_aborted` exclusion at record time). Each *completed* run appends `(predicted_s, actual_s)` to `config.ACQUISITION_HISTORY_PATH` (per-user app-data, atomic write, last `ACQUISITION_HISTORY_MAX` kept). The running batch's status line shows the same range as remaining time (`_fmt_remaining`: a range, then an "up to X left" form once the typical estimate is passed, then blank). `_begin_acq_record` captures the run context at start; `_record_acquisition` (from `_stop_elapsed`) learns + persists.
  - **Live elapsed timer + saved duration** — while an acquisition runs, a `QTimer` ticks "Elapsed: mm:ss" into that tab's time label (ground truth vs the estimate); `_start_elapsed`/`_tick_elapsed`/`_stop_elapsed` manage it, and the estimate-refresh methods skip the label while it owns it. On finish, `_stop_elapsed` appends the real elapsed to the status bar **and** `_append_elapsed_to_log` writes an `[Acquisition timing]` section (estimated vs actual) into the run's `parameters_<name>.txt`, so every dataset carries a permanent record of how long it took.
  - **Auto-stop live before acquiring** — `_stop_worker_silently(worker, finished_slot)` stops a running live feed and waits for it to release the camera, with its `finished_signal` disconnected so the queued handler can't reset the new acquisition's UI state. Called at the top of `start_zstack(camera)` for that camera's live worker, so the user never has to click Stop first. A `start_zstack` is also refused while another stack is running (one PI stage).
  - **Preset save/load + session restore** — `_collect_preset`/`_apply_preset` build/apply one versioned dict (one `orca` + one `evk4` params block, **AWG** freq/amp + VISA address, Z-stack step/num-steps + **PI focus**, the ORCA raw-stack toggle, output dirs). Older files that stored separate standalone/Z-stack copies per camera (`orca`/`orca_zstack`, `evk4`/`evk4_zstack`) still load — the Z-stack copy is preferred, falling back to the standalone one. AWG/PI restore the UI values only — the AWG output toggle is never auto-enabled (it drives hardware). New keys are optional, so older preset files still load. The manual `save_preset`/`load_preset` buttons write/read named JSON files; **additionally**, `_save_session` writes that same dict to `config.SESSION_STATE_PATH` (per-user app-data, env-overridable) and `_load_session` restores it at the end of `__init__`, so the app **starts with the last session's parameters** (silent/best-effort; filename bases stay timestamped). Persistence does **not** depend on a clean shutdown: `_save_session` runs on `closeEvent`, on `QApplication.aboutToQuit`, **and** on a 15 s periodic autosave (`_autosave_timer`) — so settings survive even an abrupt stop that skips `closeEvent` (e.g. the IDE's stop button). It builds the dict before touching the file and writes via a temp file + atomic `os.replace`, so an interrupted write can never leave an empty/corrupt session file (which would silently reset everything to defaults).
  - **Interactive crop tool** — the video feed is a `VideoFeedLabel` (see widgets) with a control bar (**Select Crop Region** / **Apply Crop** / **Reset to Full**). In crop mode the user drags a rectangle over the live image; `region_drawn` reports it in frame pixels, `_frame_region_to_sensor` maps it to absolute sensor pixels (via the active ROI origin), and `_apply_crop` converts that to the widget's width/height + centre-offset model and sets the **active camera's** params — which re-applies live via the existing `roi_changed` path. The tool is **camera-agnostic**: `_active_crop_target()` returns `(params_widget, sensor_w, sensor_h)` for whichever camera the current tab shows (ORCA or EVK4), so the same drag-to-crop works for **both** (their params widgets expose the identical `spin_roi_width/height` + `slider_offset_x/y` + `_compute_roi` ROI interface). Lets you *see* the crop before committing.
  - **`collect_acquisition_metadata`** — gathers the full instrument state (both cameras, AWG, stage) into the ordered dict that becomes the parameter log for every acquisition.
  - **Filename base** — each tab's filename field is pre-filled with a `<prefix>_<timestamp>` default (`_stamp_default_filename` records it). At the **start of an acquisition** `_refresh_default_filename` re-stamps it with the *current* time **only if it still holds that default**, so the saved files carry the acquisition time and — crucially — a name the user typed is never overwritten. The finish handlers no longer touch the field, so stopping a **live** feed (which shares the same `finished_signal`) can't wipe a typed filename. `self._fn_defaults` maps each field to its last auto-default.
  - Collapsible AWG and PI-stage sections, per-tab output-dir/filename groups, and a `closeEvent` that saves the session then tears down all workers and devices.
- **`widgets.py`** — composite, reusable control panels (one instance each: `OrcaParamsWidget` lives in the Z-Stack (ORCA) tab, `Evk4ParamsWidget` in the Z-Stack (EVK4) tab):
  - `Evk4ParamsWidget` — biases, duration, post-processing toggles, "Apply Biases to Live Feed", and an **ROI crop** (width/height + draggable centre-offset X/Y sliders, against the 1280×720 IMX636 sensor) mirroring the ORCA's ROI interface so the interactive crop tool drives it too. Changing the ROI emits a debounced `roi_changed`; while a live feed runs main_window re-crops it in real time. Unlike the ORCA's hardware subarray, the EVK4 ROI is a **software crop** of the (full-sensor) event frame and accumulated image (`core.crop_to_roi`), plus a **best-effort hardware ROI** (`apply_event_roi`, set only for fixed-window acquire/Z-stack runs to cut the event rate) — so correctness never depends on the SDK's ROI API. `get_params` adds `evk4_roi`; `get_preset`/`set_preset` persist the control values.
  - `Evk4QueueWidget` — a **batch queue** (in the Z-Stack (EVK4) tab) for running several acquisitions unattended. A table where each row is one acquisition setting (the four biases `fo/hpf/on/off` + per-plane duration) plus a **Repeats** count and a filename; **Auto-name** fills filenames from the swept parameter (`20_bias_on_off`, `10_duration`, …). It only edits the queue and emits `run_requested`/`stop_requested`/`changed`; MainWindow owns sequencing (`_start_evk4_queue` expands repeats into uniquely-named `<name>_repNN` acquisitions, then drives the existing `start_zstack("event")` path once per item, advancing on each `finished` signal) and the **total-time estimate** (`_update_queue_estimate`/`_evk4_queue_total_predicted_s` sums every row×repeat through the same calibrated EVK4 model, shown live above the Run button — these queues can run for hours). Everything not in the table (ROI, post-processing, Z-stack geometry, output dir, AWG) is inherited from the EVK4 tab at run time. The table uses the global dark stylesheet (`QTableWidget`/`QHeaderView` rules in `config.STYLESHEET`); the Filename column stretches while the short numeric columns shrink to content.
  - `OrcaParamsWidget` — exposure (min 0.017 ms, 3 decimals), frame count, **ROI** (width/height + **centre-offset X/Y as draggable `QSlider`s** with live value labels — drag / click-to-jump / scroll-wheel, no typing), mode/readout combos, a live **framerate label**, and "Apply All Settings to Live Feed". `_compute_roi` starts centred, shifts by the slider offsets, clamps to the sensor, and floors size *and* position to multiples of 4 (so the hardware subarray engages instead of silently degrading to a software crop). Changing any ROI geometry emits a **debounced `roi_changed`** signal; while a live feed runs, main_window connects it to a live-apply slot so the crop **follows the sliders in real time** (centre the crop on the speckle by dragging — note a full-sensor ROI has no room to offset, so reduce the size first). **Only the crop applies live**: exposure/mode/readout edits are pushed to the running feed only when **Apply** is clicked (`_apply_orca_roi_live` merges just the new ROI onto the last-applied settings, while `_apply_orca_params_live` / the Apply button push everything), mirroring the EVK4 (biases on Apply, crop live). `estimated_frame_time_s` / `_update_framerate` use `(Vn+1)·1H`; `estimated_raw_save_s(n)` estimates the raw-stack disk-write time and `estimated_compute_s(n)` the per-plane DSI reconstruction time. Presets persist the offsets. The **interactive crop tool** (main_window) is an alternative to dragging these sliders by hand.
  - `AWGWidget` — independent CH1/CH2 LC speckle control (VISA resource pick, **waveform type** (SQUARE/SINE/RAMP/PULSE via the SCPI `WVTP`), freq/amp, output toggles). The waveform is persisted in presets and recorded in the acquisition parameter log.
  - `PIStageWidget` — connect / move-to-focus / manual stepping / compute Z-stack start; idle position polling that can be **paused/resumed** so it doesn't fight the orchestrator for the GCS link.
  - `VideoFeedLabel` — the live-feed `QLabel` with the crop-region selector overlay. Keeps the **full-resolution** frame (`set_frame_pixmap`) and re-scales it to fit on every `resizeEvent`, so a tab switch / Display-mode change / window resize never leaves a stale, wrongly-sized frame (the old "scale once at render time" approach showed a mis-sized feed until the next frame or a manual window/fullscreen toggle). Tracks the source frame size (`set_source_size`) to map widget↔image pixels (pixmap centred + KeepAspectRatio-scaled); in crop mode it draws the dragged selection as a green box with dashed guide lines and keeps it on screen after release (so the crop can be reviewed), emitting `region_drawn(x,y,w,h)` in frame pixels. main_window owns the apply/reset logic.
- **`orchestrator.py`** — `AutomatedZStackWorker(QThread)`, the **3D integration point**. Drives the PI stage to each Z plane and acquires with the **selected camera**:
  - **ORCA**: a speckle frame stack → `compute_dsi_images` (avg + std); each plane's raw frames written to their own `<name>_raw_stack_zNNN.tif` via `save_raw_stack_tiff(..., plane=step)` (one file per plane); emits the normalized std image + `z_profile_update`.
  - **EVK4**: a fixed-duration recording per plane (device re-init per plane for clean state). The plane's raw event stream is logged to its own `<filename>_events_zNNN.raw` (one file per plane, mirroring the ORCA's per-plane raw TIFF), and the sectioned image is then reconstructed by **re-reading that raw file** via `accumulate_event_frame` (+ optional filter/smooth), **not** from the live device iteration — the live stream competes with the concurrent raw logger and drops events (occasionally whole planes → blank images), so it can't be trusted for the saved image. This mirrors the proven single-acquisition path `CameraWorker.process_final_image` in `event_camera.py`. The image is then **cropped to the ROI** (`crop_to_roi`, with a best-effort `apply_event_roi` hardware ROI per plane). `_capture_plane_event(step)` drives the device with a bounded generator (`events_for_duration`) for the acquisition window (feeding the logger; flat memory, abort discards the partial plane), stops raw logging in a `finally`, then reconstructs from the file; only when no raw file is being written does it fall back to live accumulation. Each plane's raw is kept alongside the consolidated `zstack_event_*.tif` volume.
    - **Recording watchdog (anti-freeze + no-events)** — a plane must record for ~`acqu_time` of wall-clock then stop. The normal exit is the elapsed check inside `events_for_duration`, but that only fires *after* the SDK yields a chunk. If the camera delivers **nothing** — a `bias_hpf` at its ceiling suppresses all events, or the EVK4 falls back off USB3 — `for evs in mv_iterator` blocks in native code with nothing to yield and never stops (which, with the old large ceiling, hung every plane and froze the UI). A **wall-clock watchdog thread** (`ceiling = acqu_time + 5 s`, the 5 s a grace for stream start-up / first-chunk latency) force-stops the event stream (`get_i_events_stream().stop()`, then `get_i_device_control().stop()`, both guarded) once the ceiling passes **or** the user Stops; that unblocks the iterator, so an empty (or delivery-stalled) plane still finishes in ~`acqu_time` instead of hanging, and **Stop stays responsive**. The plane is reconstructed from whatever raw was captured and the stack continues; a plane whose image is all-zero emits an explicit "0 events recorded — check the USB3 connection and lower bias_hpf" status so the cause is visible rather than just a slow blank stack. Raw logging is stopped exactly once via a lock-guarded flag shared by the watchdog and the `finally` (a double `stop_log_raw_data` can crash the native lib).
  - At the end, per-plane sectioned images are stacked into depth volumes and saved via `save_volume_tiff`, the **axial-sectioning profile** (Fig. 3a) is fitted and saved via `save_axial_sectioning_plot` (FWHM reported in the status bar), and — for ORCA — a **companion axial-average profile** of the widefield image is saved via `save_axial_average_plot` (straight-line fit, showing the average has no sectioning), plus a parameter log that records the z-positions. Each ORCA plane's raw file is written as that plane completes, so partial/aborted stacks keep the raw files of every plane that finished and still save whatever sectioned planes completed.

```
  → OrcaWorker("live") / CameraWorker("live") (thread): stream frames/events
```

```
       save_parameter_log
```

## 5. Recent work / gotchas worth remembering

- **Framerate accuracy & ROI alignment** — `ORCA_ROW_READOUT_US` now holds the manual's true `1H` values; the framerate estimate uses `(Vn+1)·1H`. Reaching the high rates the vendor software (HCImageLive) shows requires **Fast scan + small subarray height + short exposure** (the label reads "exposure-limited" when 1/exposure is the bottleneck — e.g. 50 ms ⇒ 20 fps regardless of ROI). The ROI position **must** be a multiple of 4 or DCAM silently falls back to a slow full-sensor software crop; `_compute_roi` enforces this.
- **Crop centre offset** — the ROI can be shifted off-centre (offset X/Y) to follow a laser spot that isn't centred on the sensor; the region is clamped to the sensor and 4-px-aligned, and the offsets are saved in presets.
- **Measured FPS readout** — `OrcaWorker` reports the *real* rate, not just the prediction: live mode emits "Live display: ≈ X fps" (~2×/s; this is display-limited by the preview emit), and acquisition appends "[captured N frames in T s ≈ Y fps]" to its final status line (the authoritative capture throughput). The acquire loop throttles the preview/status to ~20 updates per run so display work doesn't cap the read loop at high frame rates.
- **`process_dsi` is legacy** — the live pipeline uses `compute_dsi_images` (avg + std). Don't wire new features to `process_dsi` without reason.
- **ORCA RAM safety** — a full-sensor stack at a high frame count is gigabytes, and a Z-stack repeats it per plane, so the capture path is written to keep host-side copies to a minimum: both ORCA capture loops copy each frame **straight into one preallocated array** (no intermediate Python list + `np.array()` of it), and `compute_dsi_images` is chunked (see above). The camera's own DCAM ring buffer (`buf_alloc(N)`) still holds N frames — that floor is unavoidable. As a backstop, `start_zstack` runs a **pre-flight estimate** (`_zstack_orca_peak_bytes`) against free physical RAM (`_available_memory_bytes`, Win32 `GlobalMemoryStatusEx`) and warns before launching a run that wouldn't fit, so an over-ambitious N / ROI / step count can't silently swap the machine to a standstill. The earlier "had to restart the computer" lag came from the old path keeping ~5–7 full copies of the stack in RAM at once.

- **Per-acquisition output subfolder** — every Z-stack now writes into its own `<output_dir>/<filename>/` folder (created once at the start of `AutomatedZStackWorker.run` via `_prepare_output_dir`, which repoints `output_dir` so all save sites — per-plane raw, depth-volume TIFFs, axial profile, parameter log — land together). Keeps the many files of one acquisition self-contained; pairs with the EVK4 batch queue, whose `_repNN` filenames each get their own folder.
- **FOV matching (ORCA crop → EVK4 field)** — the Z-Stack (ORCA) tab has a "Match View to Event Camera" group with three buttons. **Match ORCA Crop to EVK4 Field…** maps the EVK4 tab's current ROI through the affine currently in force (`MainWindow._fov_affine`: the last measured/confirmed one from `config.FOV_MATCH_PATH`, else `config.EVK4_TO_ORCA_AFFINE`; `core.map_evk4_window_to_orca`) and shows a **validation preview** (`FovMatchPreviewDialog` in widgets.py: proposed crop in green, the true ~43°-rotated EVK4 footprint in magenta, drawn over the last ORCA frame placed at its ROI position via `_compose_orca_sensor_canvas`) — the crop is only applied (`OrcaParamsWidget.set_roi_window`, exact round-trip through `_compute_roi`) and persisted (`_save_fov_match` → `config.FOV_MATCH_PATH`, including the affine + method) when the user confirms. **Measure & Match (Capture Both Cameras)…** is for when the cameras were **physically moved/rotated**: it frees both cameras, runs `FovRegistrationWorker` (ui/fov_registration.py — full-sensor ORCA burst average with subarray forced OFF + a 2–5 s full-sensor EVK4 event accumulation, refusing near-empty event images with a "is the AWG running?" hint), re-measures the affine with `core.register_evk4_to_orca` (seeded by the current affine, auto-widening if weak), and previews the result with a **green/magenta registration overlay** background (`compose_registration_overlay`) + measured θ/scale/NCC (low-NCC warning below 0.5); on confirmation the new affine is adopted for future matches and persisted. The measurement counts as `busy` in `_refresh_buttons` (it owns both cameras) and is stopped in `closeEvent`. **Use Last Matching Crop** re-applies the persisted crop directly (tooltip shows geometry + confirmation date; disabled until one exists). Because the footprint is rotated, the crop is its bounding box — the green corners outside the magenta outline are ORCA-only; the dialog also warns when the footprint clips the sensor or when ORCA binning ≠ 1×1 (the calibration's condition). Applying via the spinboxes fires the normal debounced `roi_changed`, so a running live feed re-frames immediately; the 4-px alignment guarantees the **DCAM hardware subarray** engages (real framerate gain, not a software crop).
- **EVK4 batch queue** — the EVK4 tab can queue many acquisitions (varying the four biases / duration) and run them back-to-back unattended (`Evk4QueueWidget` + `_start_evk4_queue`/`_run_next_queue_item`/`_stop_evk4_queue` in MainWindow). It reuses the single-acquisition `start_zstack("event")` path per item, so validation, metadata, timing, and the output-subfolder all behave identically; the queue stops on error or via Stop, and blocks manual runs while active (`_refresh_buttons`).

## 6. Why it's structured this way

- **Testability** — the physics (`core/`) is plain arrays in / arrays out, no Qt or hardware, so DSI sectioning and event accumulation can be verified without a microscope.
- **Graceful degradation** — guarded SDK imports mean the GUI is fully buildable on any machine; missing hardware surfaces as a dialog, not a crash.
- **Responsiveness** — every blocking operation is on a worker thread; the UI thread only paints and routes signals.
- **Swappability** — adding a detector or moving math around touches one layer, because dependencies only ever point downward (`ui → hardware → core → config`).
