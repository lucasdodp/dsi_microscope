# DSI Microscope — Project Overview & Code Architecture

Control software for the **Institut Fresnel event-based Dynamic Speckle Illumination (event-DSI) microscope**. It is a PyQt6 desktop application that drives the full optical setup and reconstructs optically-sectioned wide-field fluorescence images.

---

## 1. The scientific idea

Wide-field fluorescence microscopy is fast and cheap but has **no depth discrimination**: light from out-of-focus planes blurs into the image and destroys contrast. **Dynamic Speckle Illumination (DSI)** recovers optical sectioning without a confocal pinhole or scanning.

The principle:

- The sample is illuminated with a **fluctuating random speckle pattern** (here generated by a zwitterion-doped liquid-crystal diffuser driven by an AC field).
- The detection point-spread function is sharpest **at the focal plane** and diverges away from it. So **in-focus** speckle grains are sharp and produce large pixel-to-pixel intensity fluctuations as the speckle decorrelates, while **out-of-focus** signal is spatially averaged and barely fluctuates.
- Computing the **per-pixel standard deviation** (or fluctuation) over a stack of many speckle frames therefore yields an image dominated by in-focus signal → optical sectioning (~2 µm axial).

This project implements **two acquisition modalities** of that idea:

| Modality | Detector | How sectioning is computed | Output |
|----------|----------|----------------------------|--------|
| **Traditional DSI** | Hamamatsu ORCA-Fusion sCMOS (frame-based) | Software: std-dev / fluctuation across a frame stack (`process_dsi`) | Sectioned image per Z plane |
| **Event-DSI** | Prophesee EVK4 event camera (neuromorphic) | Hardware: the sensor only fires events on intensity *changes*, so accumulating events ≈ fluctuation map, with virtually no post-processing | 2D event-count image |

The **event-DSI** advantage (per the reference papers): the sensor reacts to intensity *variations* by design, so it natively performs the operation DSI needs. Data shrinks from gigabytes of frames to tens of kB of events, with little to no compute. The trade-offs handled in code are event-camera artifacts: **"crazy pixels"** (a few pixels that self-trigger enormously) and **sensor inhomogeneity** (corrected by smoothing).

### The instruments

- **532 nm CW laser** → beam expanded onto a **liquid-crystal (LC) diffuser** → generates random speckle. The LC's electrohydrodynamic instabilities (and thus the speckle decorrelation time) are driven by a **square-wave AC field** from the AWG.
- **Siglent SDG6022X AWG** — drives the LC (frequency/amplitude set the speckle dynamics).
- **40× / 0.65 NA objective** on a **PI motorized Z-stage** for focus and Z-stacks.
- **Detection path** splits to either the **ORCA sCMOS** or the **Prophesee EVK4** event camera.

## 2. Architectural principle: strict layering

The codebase is organized into four layers with a **one-directional dependency rule**, so the physics math stays testable and the hardware stays swappable:

```
        ┌─────────────────────────────────────────────┐
        │  ui/   (PyQt6 — windows, widgets, threads)   │
        │  main_window · widgets · orchestrator        │
        └───────────────┬──────────────┬──────────────┘
                        │              │
            ┌───────────▼──────┐   ┌───▼──────────────┐
            │  hardware/       │   │  core/           │
            │  SDK wrappers +  │──▶│  pure NumPy /     │
            │  QThread workers │   │  OpenCV / SciPy   │
            └───────────┬──────┘   └───┬──────────────┘
                        │              │
                  ┌─────▼──────────────▼─────┐
                  │  config.py               │
                  │  paths · constants · CSS │
                  └──────────────────────────┘
```

**Rules enforced by the module docstrings:**

- `config.py` imports **nothing** heavy (no PyQt6, no SDKs) — safe to import everywhere.
- `core/` imports **no PyQt6 and no hardware SDK** — pure, side-effect-free math (apart from explicit `save_*` helpers). This is the optical-sectioning physics, unit-testable in isolation.
- `hardware/` wraps vendor SDKs and owns the threading. It depends on `core` (to do the math) and `config`, never on `ui`.
- `ui/` is the only layer that instantiates workers and wires Qt signals.

### Threading model

All blocking instrument I/O runs on **`QThread` workers** so the GUI never freezes. Workers communicate back to the main thread **only** via Qt signals (`status_update`, `image_ready`/`frame_ready`, `error_signal`, `finished_signal`). Every worker exposes a cooperative `stop()` that flips an `_is_running` flag checked inside its loop, and guarantees device teardown via `try/finally`.

## 3. File-by-file guide

### Entry point
- **`main.py`** — builds the `QApplication`, applies the stylesheet, shows `MainWindow`. ~20 lines.

### `config.py`
SDK paths (Hamamatsu DCAM wrapper location), hardware constants (PI serial/axis, DCAM exposure property id, EVK4 event-rate cap / fps / crazy-pixel percentile), and the dark-theme Qt stylesheet. Dependency-light by design.

### `core/` — pure processing (no Qt, no SDKs)
- **`image_processing.py`**
  - `process_dsi(stack, roi)` — the **traditional-DSI sectioning math**. Crops the (N,H,W) frame stack to the ROI, reshapes to (Y,X,N), and computes a fluctuation map as `sqrt(Σ (consecutive frame differences)²)`. Returns a scalar focus/sectioning metric `z_val` and an 8-bit normalized preview image. *(Uses true consecutive differences — no `np.roll` — to avoid wrap-around between first/last frame.)*
  - `accumulate_event_frame(chunks, w, h)` — **event-DSI image reconstruction**: sums all events per pixel via `np.add.at`. SDK-agnostic — the iterator is injected by the hardware layer.
  - `filter_crazy_pixels()` — zeros hot self-triggering pixels above a percentile.
  - `apply_smoothing()` — 5×5 Gaussian, to correct event-sensor inhomogeneity.
  - `scale_16bit_image()` — 16-bit→8-bit display scaling for ORCA live view.
  - `save_mat_tif()` — persists the final image as `.mat` + `.tif`.
- **`__init__.py`** — re-exports the above as the package's public API.

### `hardware/` — SDK wrappers + worker threads
Each SDK import is wrapped in `try/except ImportError` with an `*_AVAILABLE` flag, so the app **runs and the GUI loads even on a dev machine without the cameras/drivers** (workers emit a clean error instead of crashing).

- **`event_camera.py`** — `CameraWorker(QThread)` for the Prophesee EVK4 (Metavision SDK). Sets biases (`bias_fo`, `bias_hpf`, `bias_diff_on/off`), enables the event-rate controller, streams events into a `PeriodicFrameGenerationAlgorithm` for live view, optionally logs raw `.raw` data, and in *acquire* mode reconstructs/filters/smooths/saves the final 2D image by delegating to `core`.
- **`orca_camera.py`** — `OrcaWorker(QThread)` for the Hamamatsu ORCA via DCAM. Extends `sys.path` to the SDK sample wrapper, runs the continuous live-focus loop, guarantees buffer release / device close.
- **`stage_control.py`** — `StageController` (owns the PI `GCSDevice` handle, USB connect, FNL referencing, servo enable, position queries) and `PIMoveWorker(QThread)` (offloads blocking `MOV` + `waitontarget` moves).
- **`awg_control.py`** — `AWGController`, a thin PyVISA/SCPI wrapper for the Siglent AWG (list resources, connect, set square-wave freq/amp, toggle output). No Qt.

### `ui/` — presentation & coordination
- **`main_window.py`** — `MainWindow`. The single place that builds the three tabs (**Event Camera 2D**, **Scientific Camera ORCA**, **3D Z-Stack**), instantiates the workers, routes every worker's `status_update`/`error_signal` to one status bar, manages button enable/disable during acquisition, collects UI parameters into dicts (`get_evk4_params`, `get_orca_params`, `get_motor_params`), and renders frames in `update_image` (NumPy → `QImage` → `QPixmap`). `closeEvent` tears down all workers and devices.
- **`widgets.py`** — composite control panels that own their visuals and delegate I/O to the hardware controllers: `AWGWidget` (LC speckle control) and `PIStageWidget` (connect / move-to-focus / manual stepping / compute Z-stack start position).
- **`orchestrator.py`** — `AutomatedZStackWorker(QThread)`, the **3D event-DSI integration point**. It is the one component that composes *two* instruments plus the math: it drives the PI stage to each Z plane, captures an ORCA frame stack there, runs `process_dsi` on it, and emits the sectioned image + Z-profile value per step. It lives in `ui/` (not `hardware/`) deliberately, to avoid a `hardware → core → hardware` import tangle.

## 4. Typical data flows

**Event-DSI 2D acquisition** (Event Camera tab):
```
MainWindow.start_evk4("acquire")
  → CameraWorker (thread): set biases → stream events → log .raw
  → process_final_image:  EventsIterator
                          → core.accumulate_event_frame   (events → image)
                          → core.filter_crazy_pixels
                          → core.apply_smoothing
                          → core.save_mat_tif
  → status/image signals → MainWindow status bar + video feed
```

**Automated 3D DSI Z-stack** (3D Z-Stack tab):
```
MainWindow.start_zstack
  → AutomatedZStackWorker (thread):
       for each Z step:
         PI stage MOV → waitontarget → settle
         ORCA: buf_alloc → cap_start → grab N frames
         core.process_dsi(stack, roi) → (z_val, sectioned image)
         emit image_ready + z_profile_update
  → MainWindow updates feed; z-profile printed per step
```

---

## 5. Why it's structured this way

- **Testability** — the physics (`core/`) is plain arrays in / arrays out, no Qt or hardware, so DSI sectioning and event accumulation can be verified without a microscope.
- **Graceful degradation** — guarded SDK imports mean the GUI is fully buildable on any machine; missing hardware surfaces as a status error, not a crash.
- **Responsiveness** — every blocking operation is on a worker thread; the UI thread only paints and routes signals.
- **Swappability** — adding a detector or moving math around touches one layer, because dependencies only ever point downward (`ui → hardware → core → config`).
