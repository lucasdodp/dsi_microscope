# DSI Microscope — Project Overview & Code Architecture

Control software for the **Institut Fresnel event-based Dynamic Speckle Illumination (event-DSI) microscope**. It is a PyQt6 desktop application that drives the full optical setup and reconstructs optically-sectioned wide-field fluorescence images.

> This document is the high-level map kept for context across sessions. When you change behaviour, update the relevant section here too.

---

## 1. The scientific idea

Wide-field fluorescence microscopy is fast and cheap but has **no depth discrimination**: light from out-of-focus planes blurs into the image and destroys contrast. **Dynamic Speckle Illumination (DSI)** recovers optical sectioning without a confocal pinhole or scanning.

The principle:

- The sample is illuminated with a **fluctuating random speckle pattern** (here generated by a zwitterion-doped liquid-crystal diffuser driven by an AC field).
- The detection point-spread function is sharpest **at the focal plane** and diverges away from it. So **in-focus** speckle grains are sharp and produce large pixel-to-pixel intensity fluctuations as the speckle decorrelates, while **out-of-focus** signal is spatially averaged and barely fluctuates.
- Computing the **per-pixel standard deviation** (or fluctuation) over a stack of many speckle frames therefore yields an image dominated by in-focus signal → optical sectioning (~2 µm axial). The **per-pixel average** of the same stack is the conventional widefield-equivalent image.

This project implements **two acquisition modalities** of that idea:

| Modality | Detector | How sectioning is computed | Output |
|----------|----------|----------------------------|--------|
| **Traditional DSI** | Hamamatsu ORCA-Fusion BT sCMOS (frame-based) | Software: per-pixel std-dev across a frame stack (`compute_dsi_images`) | Average (widefield) + std-dev (sectioned) image |
| **Event-DSI** | Prophesee EVK4 event camera (neuromorphic) | Hardware: the sensor only fires events on intensity *changes*, so accumulating events ≈ fluctuation map, with virtually no post-processing | 2D event-count image |

The **event-DSI** advantage (per the reference papers): the sensor reacts to intensity *variations* by design, so it natively performs the operation DSI needs. Data shrinks from gigabytes of frames to tens of kB of events, with little to no compute. The trade-offs handled in code are event-camera artifacts: **"crazy pixels"** (a few pixels that self-trigger enormously) and **sensor inhomogeneity** (corrected by smoothing).

### The instruments

- **532 nm CW laser** → beam expanded onto a **liquid-crystal (LC) diffuser** → generates random speckle. The LC's electrohydrodynamic instabilities (and thus the speckle decorrelation time) are driven by a **square-wave AC field** from the AWG.
- **Siglent SDG6022X AWG** — drives the LC; two independent channels (CH1/CH2). Frequency/amplitude set the speckle dynamics.
- **40× / 0.65 NA objective** on a **PI E-709 piezo Z-stage** for focus and Z-stacks.
- **Detection path** splits to either the **ORCA sCMOS** or the **Prophesee EVK4** event camera.

## 2. Architectural principle: strict layering

The codebase is organized into four layers with a **one-directional dependency rule**, so the physics math stays testable and the hardware stays swappable:

```
        ┌─────────────────────────────────────────────┐
        │  ui/   (PyQt6 — windows, widgets, threads)   │
        │  main_window · widgets · orchestrator        │
        └───────────────┬──────────────┬──────────────┘
                        │              │
            ┌───────────▼──────┐   ┌───▼──────────────┐
            │  hardware/       │   │  core/           │
            │  SDK wrappers +  │──▶│  pure NumPy /    │
            │  QThread workers │   │  OpenCV / SciPy  │
            └───────────┬──────┘   └───┬──────────────┘
                        │              │
                  ┌─────▼──────────────▼─────┐
                  │  config.py               │
                  │  paths · constants · CSS │
                  └──────────────────────────┘
```

**Rules enforced by the module docstrings:**

- `config.py` imports **nothing** heavy (no PyQt6, no SDKs) — safe to import everywhere.
- `core/` imports **no PyQt6 and no hardware SDK** — pure, side-effect-free math (apart from explicit `save_*` helpers). This is the optical-sectioning physics, unit-testable in isolation.
- `hardware/` wraps vendor SDKs and owns the threading. It depends on `core` (to do the math) and `config`, never on `ui`.
- `ui/` is the only layer that instantiates workers and wires Qt signals. The Z-stack **orchestrator** lives here (not in `hardware/`) deliberately, because it composes *two* instruments plus `core`, and putting it in `hardware/` would create a `hardware → core → hardware` tangle.

### Threading model

All blocking instrument I/O runs on **`QThread` workers** so the GUI never freezes. Workers communicate back to the main thread **only** via Qt signals (`status_update`, `image_ready`/`frame_ready`, `z_profile_update`, `position_update`, `error_signal`, `finished_signal`). Every worker exposes a cooperative `stop()` that flips an `_is_running` flag checked inside its loop, and guarantees device teardown via `try/finally`.

### Graceful degradation

Every SDK import (`dcam`, `metavision_*`, `pipython`) is wrapped in `try/except ImportError` with an `*_AVAILABLE` flag. The GUI builds and runs on a dev machine with **no cameras/drivers**; attempting an action without the SDK shows a clear dialog (with setup hints) instead of crashing.

## 3. File-by-file guide

### Entry point
- **`main.py`** — builds the `QApplication`, applies the stylesheet, shows `MainWindow`. ~20 lines.

### `config.py`
Dependency-light constants:
- **SDK path** — `HAMAMATSU_SDK_PATH` (where `dcam.py` lives), overridable by env var.
- **PI stage** — controller name, serial, RS-232 port/baud, auto-zero flag. Most are env-overridable so the same checkout runs on laptop and lab PC.
- **ORCA / DCAM** — property ids (exposure, readout speed, binning, trigger source/mode, defect correct, and the **subarray** HPOS/HSIZE/VPOS/VSIZE/MODE ids); sensor size `ORCA_SENSOR_WIDTH/HEIGHT = 2304`; `*_OPTIONS` maps ({label: dcam enum value}) surfaced in the UI combos; and **`ORCA_ROW_READOUT_US`** — the per-row readout time `1H` for each scan speed, taken directly from the camera manual (Fast 4.86765 µs, Standard 18.64706 µs, Ultra Quiet 80.0 µs). The free-running frame time is `(Vn+1)·1H`, so reducing subarray **height** is what raises the framerate.
- **EVK4** — event-rate-controller cap, display fps, crazy-pixel percentile.
- **`STYLESHEET`** — the dark-theme Qt CSS.

### `core/` — pure processing (no Qt, no SDKs)
**`image_processing.py`** — public API re-exported by `core/__init__.py`:

*ORCA / DSI:*
- `compute_dsi_images(stack, roi=None)` — **the active single-plane DSI reconstruction**. Returns the per-pixel **average** (widefield-equivalent) and **standard-deviation** (optically-sectioned) float32 images across the (N,H,W) stack. Used by both the ORCA single-Z acquire and the Z-stack. **Memory-bounded:** mean and variance are accumulated over the frame axis in float64 **chunks** (a two-pass population variance, `ddof=0`, matching the old `np.std`), instead of materializing `stack.astype(float32)` and `np.std`'s internal deviation array — each the full N×H×W size. The working set is a few H×W planes (`_DSI_CHUNK_BYTES ≈ 128 MB`), so peak RAM is independent of N (this is what stops a big stack from swapping the machine).
- `process_dsi(stack, roi)` — *legacy* consecutive-difference fluctuation estimator `sqrt(Σ(frame[i+1]−frame[i])²)`, returning a scalar focus metric + 8-bit preview. Still exported, but **no current caller** drives the pipeline with it (the Z-stack profile value is `np.sum(std_img)` from `compute_dsi_images`). Keep for reference / re-processing experiments.
- `normalize_to_8bit(img)` — min-max → uint8 for display/preview.
- `save_dsi_results(avg, std, dir, name)` — writes `average_*` and `dsi_*` as full-precision `.mat` **and** 8-bit `.tif`.
- `scale_16bit_image(data)` — 16→8-bit live-view scaling (integer stretch by current frame max, then down-shift).

*Raw / volume persistence (TIFF):*
- `save_raw_stack_tiff(stack, dir, name, roi=None)` — one plane's raw 16-bit frames as a multi-page TIFF (`raw_stack_<name>.tif`), native bit depth preserved. Used by the single-Z acquire **and** by the Z-stack, which calls it once per plane with a z-indexed name (`raw_stack_<name>_zNNN.tif`) so each plane is a **separate** raw file — the downstream MATLAB algorithms (e.g. **RIM**) consume the planes individually, not as one combined volume.
- `save_volume_tiff(volume, dir, name, kind)` — a (Z,H,W) depth stack as a multi-page TIFF (float32 → 32-bit, no per-slice normalization).
- `save_axial_sectioning_plot(z_positions, intensities, dir, name, kind)` — the **axial-sectioning profile** (paper Fig. 3a): per-plane mean sectioned-image intensity vs axial position, Gaussian-fitted (`scipy.optimize.curve_fit`) to extract the **FWHM = axial sectioning**. Always writes `axial_profile_<kind>_<name>.csv` (z, raw + peak-normalized intensity, fit params); writes the PNG only if **matplotlib** is installed (lazy OO `FigureCanvasAgg` import, thread-safe, no window). Returns `(fwhm, csv_path, png_path)`. Helpers `_gaussian`, `_fit_axial_gaussian`, `_save_axial_figure` degrade to data-only on any failure (no SciPy/matplotlib, <4 planes, non-convergence).
- `_write_multipage_tiff` / `_prepare_raw_frames` — shared internals (ROI crop + native-depth coercion).

*Logging:*
- `save_parameter_log(dir, name, sections)` — human-readable `parameters_*.txt` from an ordered `{section: {key: value}}` dict.

*EVK4:*
- `accumulate_event_frame(chunks, w, h)` — sum events per pixel via `np.add.at` (SDK-agnostic; iterator injected by hardware).
- `filter_crazy_pixels(img)` — zero hot self-triggering pixels above a percentile.
- `apply_smoothing(img)` — 5×5 Gaussian (event-sensor inhomogeneity).
- `save_mat_tif(img, dir, name)` — final EVK4 image as `.mat` + `.tif`.

### `hardware/` — SDK wrappers + worker threads
- **`orca_camera.py`** — `OrcaWorker(QThread)` for the Hamamatsu ORCA via DCAM. Two modes:
  - **live** — continuous 3-frame-ring-buffer focus loop. Supports **live parameter updates** via `apply_params()`: on the next frame it stops capture, re-applies exposure/mode/ROI, and restarts (a brief blink, far cheaper than a worker restart).
  - **acquire** — single-Z DSI: grabs N speckle frames into a buffer, then `compute_dsi_images` → saves the avg + std images, optionally the **raw 16-bit stack TIFF**, and a parameter log.
  - `_apply_camera_settings` pushes readout speed / binning / trigger / defect-correct (rejections surface as status warnings, never abort). `_apply_subarray` configures the **DCAM hardware subarray (ROI)**: MODE must be OFF before changing position/size; HPOS/HSIZE/VPOS/VSIZE must be multiples of 4 and within the sensor. **If the camera rejects the subarray, it falls back to a software crop** (`_hw_roi_active=False`) — correct image, but no framerate gain. Hardware subarray already delivers the cropped frame, so software crop is only applied when the ROI was rejected.
- **`event_camera.py`** — `CameraWorker(QThread)` for the Prophesee EVK4 (Metavision SDK). Sets biases (`bias_fo`, `bias_hpf`, `bias_diff_on/off`), enables the event-rate controller, streams events into a `PeriodicFrameGenerationAlgorithm` for live view, supports live bias updates via `apply_biases()`, optionally logs raw `.raw`, and in *acquire* mode reconstructs/filters/smooths/saves the final 2D image by delegating to `core`. Raw logging is always stopped in a `finally` (calling stop without a prior `log_raw_data` can crash the native lib). Also exports `initiate_device` / `EventsIterator` / `METAVISION_AVAILABLE` for the orchestrator to reuse.
- **`stage_control.py`** — `StageController` owns the PI `GCSDevice` handle and connection lifecycle: tries **native USB** first, then **RS-232** (forced port or auto-scan, FTDI/US232R adapters probed first). The E-709 has **no reference switches** → no FNL/FRF; it just enables the servo (`SVO`) for closed-loop, optional auto-zero (`ATZ`). Reads position unit + travel limits (`qPUN`/`qTMN`/`qTMX`) best-effort. `PIMoveWorker(QThread)` offloads blocking `MOV` + `waitontarget`, always emitting `finished_signal` (even on error) so the UI re-enables.
- **`awg_control.py`** — `AWGController`, a thin PyVISA/SCPI wrapper for the Siglent AWG. Lists resources, connects, sets per-channel (`C1`/`C2`) square-wave freq/amp, toggles output, and on `close()` forces both outputs off. No Qt.

### `ui/` — presentation & coordination
- **`main_window.py`** — `MainWindow`. The single place that builds the three tabs (**Event Camera 2D**, **Scientific Camera ORCA**, **3D Z-Stack**), instantiates all workers + the Z-stack orchestrator, routes every worker's `status_update`/`error_signal` to one status bar, manages button enable/disable during acquisition, and renders frames in `update_image` (NumPy → `QImage` → `QPixmap`, grayscale or BGR888). Also owns:
  - **Acquisition time estimates** per tab (`_update_orca_time`, `_update_zstack_time`, …): capture (`N·frame_time`) **plus** camera init, per-plane motor move/settle, the EVK4 per-plane re-init, the **per-plane DSI reconstruction** (`OrcaParamsWidget.estimated_compute_s`), and raw-stack disk write (constants in `config.py`: `ORCA_CAMERA_INIT_S`, `ORCA_PLANE_OVERHEAD_S`, `EVK4_PLANE_OVERHEAD_S`, `ORCA_DSI_PROCESS_PIXELS_PER_S`, `ZSTACK_DISK_BYTES_PER_S`). The compute term was the big missing cost: the avg+std reconstruction is ~5–6 s **per plane** at full sensor (measured ~90–100 Mpx/s), so omitting it made full-sensor stacks badly under-estimated. These are deliberately rough; the live elapsed timer is the ground truth.
  - **Live elapsed timer** — while an acquisition runs, a `QTimer` ticks "Elapsed: mm:ss" into that tab's time label (ground truth vs the estimate); `_start_elapsed`/`_tick_elapsed`/`_stop_elapsed` manage it, and the estimate-refresh methods skip the label while it owns it.
  - **Auto-stop live before acquiring** — `_stop_worker_silently(worker, finished_slot)` stops a running live feed and waits for it to release the camera, with its `finished_signal` disconnected so the queued handler can't reset the new acquisition's UI state. Called at the top of the acquire path in `start_orca`/`start_evk4`/`start_zstack`, so the user never has to click Stop first.
  - **Preset save/load + session restore** — `_collect_preset`/`_apply_preset` build/apply one versioned dict (ORCA + EVK4 params, **AWG** freq/amp + VISA address, Z-stack camera/step/num-steps + **PI focus**, save-raw toggles, output dirs), applied to *both* the standalone and Z-stack copies of each params widget. AWG/PI restore the UI values only — the AWG output toggle is never auto-enabled (it drives hardware). New keys are optional, so older preset files still load. The manual `save_preset`/`load_preset` buttons write/read named JSON files; **additionally**, `_save_session` writes that same dict to `config.SESSION_STATE_PATH` (per-user app-data, env-overridable) and `_load_session` restores it at the end of `__init__`, so the app **starts with the last session's parameters** (silent/best-effort; filename bases stay timestamped). Persistence does **not** depend on a clean shutdown: `_save_session` runs on `closeEvent`, on `QApplication.aboutToQuit`, **and** on a 15 s periodic autosave (`_autosave_timer`) — so settings survive even an abrupt stop that skips `closeEvent` (e.g. the IDE's stop button). It builds the dict before touching the file and writes via a temp file + atomic `os.replace`, so an interrupted write can never leave an empty/corrupt session file (which would silently reset everything to defaults).
  - **Interactive crop tool** — the video feed is a `VideoFeedLabel` (see widgets) with a control bar (**Select Crop Region** / **Apply Crop** / **Reset to Full**). In crop mode the user drags a rectangle over the full-sensor live image; `region_drawn` reports it in frame pixels, `_frame_region_to_sensor` maps it to absolute sensor pixels (via the active ROI origin), and `_apply_crop` converts that to the widget's width/height + centre-offset model and sets the **active** ORCA params (`_active_orca_params`: the Z-stack copy when that tab shows ORCA, else the standalone) — which re-applies live via the existing `roi_changed` path. Lets you *see* the crop before committing.
  - **`collect_acquisition_metadata`** — gathers the full instrument state (both cameras, AWG, stage) into the ordered dict that becomes the parameter log for every acquisition.
  - **Filename base** — each tab's filename field is pre-filled with a `<prefix>_<timestamp>` default (`_stamp_default_filename` records it). At the **start of an acquisition** `_refresh_default_filename` re-stamps it with the *current* time **only if it still holds that default**, so the saved files carry the acquisition time and — crucially — a name the user typed is never overwritten. The finish handlers no longer touch the field, so stopping a **live** feed (which shares the same `finished_signal`) can't wipe a typed filename. `self._fn_defaults` maps each field to its last auto-default.
  - A collapsible AWG section, three output-dir/filename groups, and a `closeEvent` that saves the session then tears down all workers and devices.
- **`widgets.py`** — composite, reusable control panels (each instantiated **twice**: once in its own tab, once in the Z-stack tab, so the two have fully independent parameters):
  - `Evk4ParamsWidget` — biases, duration, post-processing toggles, "Apply Biases to Live Feed"; `get_params`/`get_preset`/`set_preset`.
  - `OrcaParamsWidget` — exposure (min 0.017 ms, 3 decimals), frame count, **ROI** (width/height + **centre-offset X/Y as draggable `QSlider`s** with live value labels — drag / click-to-jump / scroll-wheel, no typing), mode/readout combos, a live **framerate label**, and "Apply All Settings to Live Feed". `_compute_roi` starts centred, shifts by the slider offsets, clamps to the sensor, and floors size *and* position to multiples of 4 (so the hardware subarray engages instead of silently degrading to a software crop). Changing any ROI geometry emits a **debounced `roi_changed`** signal; while a live feed runs, main_window connects it to the live-apply slot so the crop **follows the sliders in real time** (centre the crop on the speckle by dragging — note a full-sensor ROI has no room to offset, so reduce the size first). `estimated_frame_time_s` / `_update_framerate` use `(Vn+1)·1H`; `estimated_raw_save_s(n)` estimates the raw-stack disk-write time and `estimated_compute_s(n)` the per-plane DSI reconstruction time. Presets persist the offsets. The **interactive crop tool** (main_window) is an alternative to dragging these sliders by hand.
  - `AWGWidget` — independent CH1/CH2 LC speckle control (VISA resource pick, freq/amp, output toggles).
  - `PIStageWidget` — connect / move-to-focus / manual stepping / compute Z-stack start; idle position polling that can be **paused/resumed** so it doesn't fight the orchestrator for the GCS link.
  - `VideoFeedLabel` — the live-feed `QLabel` with the crop-region selector overlay. Tracks the source frame size (`set_source_size`) to map widget↔image pixels (pixmap centred + KeepAspectRatio-scaled); in crop mode it draws the dragged selection as a green box with dashed guide lines and keeps it on screen after release (so the crop can be reviewed), emitting `region_drawn(x,y,w,h)` in frame pixels. main_window owns the apply/reset logic.
- **`orchestrator.py`** — `AutomatedZStackWorker(QThread)`, the **3D integration point**. Drives the PI stage to each Z plane and acquires with the **selected camera**:
  - **ORCA**: a speckle frame stack → `compute_dsi_images` (avg + std); each plane's raw frames written to their own `raw_stack_<name>_zNNN.tif` via `save_raw_stack_tiff` (one file per plane); emits the normalized std image + `z_profile_update`.
  - **EVK4**: a fixed-duration recording per plane (device re-init per plane for clean state), accumulated **directly from the live stream** via `accumulate_event_frame` (+ optional filter/smooth). **No per-plane `.raw` file is written** — the stack's only data output is the single consolidated `zstack_event_*.tif` volume (the "save raw" checkbox is ORCA-only). `_capture_plane_event(step)` uses a bounded generator (`events_for_duration`) so memory stays flat and an abort discards the partial plane.
  - At the end, per-plane sectioned images are stacked into depth volumes and saved via `save_volume_tiff`, the **axial-sectioning profile** (Fig. 3a) is fitted and saved via `save_axial_sectioning_plot` (FWHM reported in the status bar), plus a parameter log that records the z-positions. Each ORCA plane's raw file is written as that plane completes, so partial/aborted stacks keep the raw files of every plane that finished and still save whatever sectioned planes completed.

## 4. Typical data flows

**ORCA single-Z DSI** (Scientific Camera tab → *Start DSI Acquisition*):
```
MainWindow.start_orca("acquire")
  → OrcaWorker (thread): apply exposure/mode/subarray-ROI
       buf_alloc(N) → cap_start → grab N speckle frames
       core.compute_dsi_images(stack, roi) → (avg, std)
       emit std preview
       core.save_raw_stack_tiff (optional) + save_dsi_results + save_parameter_log
  → status/image signals → MainWindow status bar + video feed
```

**Event-DSI 2D** (Event Camera tab → *Start Acquisition*):
```
MainWindow.start_evk4("acquire")
  → CameraWorker (thread): set biases → stream events → log .raw
  → process_final_image:  EventsIterator
                          → core.accumulate_event_frame
                          → core.filter_crazy_pixels / apply_smoothing
                          → core.save_mat_tif (+ save_parameter_log)
```

**Automated 3D Z-stack** (3D Z-Stack tab, either camera):
```
MainWindow.start_zstack  (pauses PIStageWidget polling; hands stage to orchestrator)
  → AutomatedZStackWorker (thread):
       move to start; for each Z step:
         MOV → waitontarget → settle (~0.5 s)
         ORCA:  grab N frames → compute_dsi_images → save_raw_stack_tiff(per-plane raw_stack_..._zNNN.tif)
         EVK4:  stream events for acqu_time → accumulate_event_frame (no per-plane .raw)
         emit image_ready + z_profile_update + position_update
       save_volume_tiff(depth volumes)
       save_axial_sectioning_plot(z, per-plane mean intensity)  # Fig. 3a: CSV + Gaussian FWHM + PNG
       save_parameter_log
```

## 5. Recent work / gotchas worth remembering

- **Framerate accuracy & ROI alignment** — `ORCA_ROW_READOUT_US` now holds the manual's true `1H` values; the framerate estimate uses `(Vn+1)·1H`. Reaching the high rates the vendor software (HCImageLive) shows requires **Fast scan + small subarray height + short exposure** (the label reads "exposure-limited" when 1/exposure is the bottleneck — e.g. 50 ms ⇒ 20 fps regardless of ROI). The ROI position **must** be a multiple of 4 or DCAM silently falls back to a slow full-sensor software crop; `_compute_roi` enforces this.
- **Crop centre offset** — the ROI can be shifted off-centre (offset X/Y) to follow a laser spot that isn't centred on the sensor; the region is clamped to the sensor and 4-px-aligned, and the offsets are saved in presets.
- **Measured FPS readout** — `OrcaWorker` reports the *real* rate, not just the prediction: live mode emits "Live display: ≈ X fps" (~2×/s; this is display-limited by the preview emit), and acquisition appends "[captured N frames in T s ≈ Y fps]" to its final status line (the authoritative capture throughput). The acquire loop throttles the preview/status to ~20 updates per run so display work doesn't cap the read loop at high frame rates.
- **`process_dsi` is legacy** — the live pipeline uses `compute_dsi_images` (avg + std). Don't wire new features to `process_dsi` without reason.
- **ORCA RAM safety** — a full-sensor stack at a high frame count is gigabytes, and a Z-stack repeats it per plane, so the capture path is written to keep host-side copies to a minimum: both ORCA capture loops copy each frame **straight into one preallocated array** (no intermediate Python list + `np.array()` of it), and `compute_dsi_images` is chunked (see above). The camera's own DCAM ring buffer (`buf_alloc(N)`) still holds N frames — that floor is unavoidable. As a backstop, `start_zstack` runs a **pre-flight estimate** (`_zstack_orca_peak_bytes`) against free physical RAM (`_available_memory_bytes`, Win32 `GlobalMemoryStatusEx`) and warns before launching a run that wouldn't fit, so an over-ambitious N / ROI / step count can't silently swap the machine to a standstill. The earlier "had to restart the computer" lag came from the old path keeping ~5–7 full copies of the stack in RAM at once.

## 6. Why it's structured this way

- **Testability** — the physics (`core/`) is plain arrays in / arrays out, no Qt or hardware, so DSI sectioning and event accumulation can be verified without a microscope.
- **Graceful degradation** — guarded SDK imports mean the GUI is fully buildable on any machine; missing hardware surfaces as a dialog, not a crash.
- **Responsiveness** — every blocking operation is on a worker thread; the UI thread only paints and routes signals.
- **Swappability** — adding a detector or moving math around touches one layer, because dependencies only ever point downward (`ui → hardware → core → config`).
